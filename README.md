# bim-database

A small catalog of residential buildings (prefab products + scanned
documentation of real existing houses), shaped as the **lighthouse corpus
for [`bim-ai`](https://github.com/jhoetter/bim-ai)** — a system that
generates BIM models from architectural drawings.

Every house here is annotated with a structured ontology *and* a live
verdict on whether `bim-ai` can model it correctly today. When a house
exposes a capability gap, a GitHub issue is filed against `bim-ai` and
linked from the record; once the issue closes, the badge flips to green
on the next state refresh.

## Quick start

```bash
make install     # creates .venv, installs deps
make dev         # starts FastAPI on http://localhost:2500 (with --reload)
```

Open <http://localhost:2500> — single-page UI, slice the catalog by source,
type, roof, style, era, area, price, energy standard, or current `bim-ai`
modelability.

Other targets:

```bash
make validate              # JSON schema + ontology + image-existence checks
make new-house ID=72 MODEL="…"   # scaffold a stub record + folder
make refresh-issue-state   # poll GitHub for each linked issue's open/closed state
make warm-cache            # pre-render every PDF-sourced scene into tmp/scene-cache/
make mcp                   # stdio MCP server for Claude Desktop / agents
```

## What's in the box

```
data/
  ontology.json                # single source of truth for every enum (roof types,
                               # styles, image categories/mediums, …)
  .issue_state.json            # cached open/closed state of each linked bim-ai issue
                               # (refreshed by `make refresh-issue-state`)
  houses/
    house-1/
      house-1.json             # the record itself (schema/house.schema.json)
      house-1.pdf              # combined-overview PDF (auto-generated by build_houses.py)
      *.avif                   # exterior + floorplan images (catalog originals)
      *.pdf                    # source Bauplane / Baubeschreibung (documentation houses)
    house-2/
      …
  # NB: house-N-{category}-{label}.jpg crops are NOT committed — they're
  # reconstructed on demand by api/scene_render.py from the JSON source_ref
  # coords + source PDFs. See "Scene reconstruction" below.
schema/
  house.schema.json            # JSON Schema for one house record
api/
  main.py                      # FastAPI: /houses, /ontology, /static, /scene
  scene_render.py              # render-on-demand engine for PDF-sourced crops
mcp_server.py                  # MCP server with the same data model
ui/index.html                  # single-page UI, filters built from /ontology
build_houses.py                # walks data/houses/*/ → generates missing house-N.pdf
scripts/
  new_house.py                 # `make new-house` scaffolder
  validate.py                  # `make validate`
  refresh_issue_state.py       # `make refresh-issue-state`
  render_scene.py              # CLI for the render cache (one-off, --all warm)
spec/
  scene-image-storage.md       # storage tracker: how / why crops aren't in git
AGENTS.md                      # how to add a house, fill in ontology, file bim-ai issues
tmp/
  scene-cache/                 # rendered AVIF crops (gitignored — rebuilt on demand)
```

The numeric ID is the spine: it appears as the folder name (`house-N/`),
the metadata file (`house-N.json`), and the combined PDF (`house-N.pdf`).
`make validate` enforces the chain.

## The bim-ai modelability flag

Each record carries an optional `bim_ai_blocking_issues` field — a list of
`owner/repo#NNN` GitHub-issue references that currently prevent `bim-ai`
from rendering the house correctly. The API derives a tri-state badge from
the cached issue state:

| `bim_ai_blocking_issues` value | API result | UI badge |
|---|---|---|
| field absent / `null`         | not assessed         | _(no badge)_ |
| `[]`                          | assessed clean       | green **BIM-AI ✓** |
| has open issue ref            | currently blocked    | red **BIM-AI ✗** |
| has uncached ref              | refresh needed       | grey **BIM-AI ?** |

State comes from `data/.issue_state.json`, which `make refresh-issue-state`
polls via the `gh` CLI. UI filter `bim-ai: ✓ modellierbar / ✗ blockiert`
slices the catalog by this signal.

A working example: `house-18` (Fertighaus WEISS *MH STYLE*) is a
Sonnenenergiehaus with a *Versetztes Pultdach* + solar clerestory band.
Originally blocked by `jhoetter/bim-ai#101` (offset-pultdach roof mode);
the issue was implemented and closed, so the badge is now green.

See **AGENTS.md** for the detailed walkthrough on assessing a new house
and filing a capability-gap issue (screenshot conventions, `assets-v1`
release as the inline-image bucket, etc.).

## Adding a new house

```bash
make new-house ID=72 MODEL="My new house"
# drops a stub at data/houses/house-72/house-72.json + an empty folder
# next: drop AVIF photos into data/houses/house-72/, then edit the JSON

make validate                         # checks schema + ontology
```

Code reads [CLAUDE.md](CLAUDE.md) which points to the same file.

## Scene reconstruction (documentation houses)

For *documentation* houses (h21, h22, h23, …) where the source is a scanned
multi-page Bauplan, individual scene crops (one elevation, one floorplan, one
section per artifact) are **reconstructed on demand** rather than committed
as JPGs. Only the JSON coords are tracked.

```
data/houses/house-22/
  Grundrisse, Ansichten, Schnitt (1).pdf   # the source (committed, immutable)
  house-22.json                            # one entry per scene with source_ref:
                                           #   { file, page, crop_box_pct,
                                           #     rotation_deg, dpi, page_title, scale }
```

When the UI requests `/scene/house-22/house-22-elevation-nord.jpg`, the API:

1. looks up the scene's `source_ref` in `house-22.json`
2. renders PDF page `5` at 200 dpi
3. rotates by `rotation_deg` (counter-clockwise)
4. crops by `crop_box_pct` `[x0,y0,x1,y1]` (post-rotation)
5. caches as `tmp/scene-cache/house-22/house-22-elevation-nord.avif` (gitignored)
6. serves with `Content-Type: image/avif`

Cold render ~210 ms, warm cache ~4 ms. The cache invalidates automatically
when `house-22.json` is newer than the cached image.

The end-to-end agent workflow that produces these `source_ref` entries —
inventory the PDFs, decompose multi-artifact pages, iterate crops until
they're neither tight nor bleeding into neighbors, then annotate facts —
lives in the [house-data-standardization skill][skill] in `bim-agent`.
The storage strategy behind the render cache (why this isn't just JPGs in
git) is documented in [`spec/scene-image-storage.md`](spec/scene-image-storage.md).

[skill]: https://github.com/jhoetter/bim-agent/blob/main/claude-skills/house-data-standardization/SKILL.md

Catalog houses (h1-20, h24-71) keep their original AVIFs committed and serve
through `/static/`; the render-cache path is only for PDF-sourced crops.

## REST API

| route | returns |
|---|---|
| `GET /` | the single-page UI |
| `GET /houses` | every record (filter by `source`, `building_type`, `construction`, `roof_type`, `style`, `energy_standard`, `has_basement`, `min_area`, `max_area`, `max_price`, `min_year`, `max_year`, `modelable_in_bim_ai`) |
| `GET /houses/{key}` | one record by `house-N` key (bare `N` also accepted) |
| `GET /houses/{key}/pdf` | combined PDF |
| `GET /houses/{key}/images` | image-metadata array |
| `GET /ontology` | the enum vocabulary (UI dropdowns are built from this) |
| `GET /static/{path}` | per-house assets that live in git (AVIFs, source PDFs) |
| `GET /scene/{key}/{file}` | PDF-sourced scene crops, rendered on demand and cached as AVIF (see *Scene reconstruction* above) |

## MCP server

`make mcp` exposes the same data model to Claude Desktop / Claude Code /
any MCP-aware agent:

| tool | use case |
|---|---|
| `get_ontology()`         | discover valid filter values before querying |
| `get_database_summary()` | shape of the catalog (counts by source / type / roof) |
| `list_houses(...)`       | same filters as `GET /houses`, returns records with image URLs |
| `get_house(key)`         | full record by key |

The MCP server uses `http://localhost:2500` as the static-asset base, so
`make dev` should be running for image / PDF URLs to resolve.
